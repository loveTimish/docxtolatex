# IMPLEMENTATION_PLAN.md

本文档是一版 **待审核的实施计划**。

目标不是一下子把所有功能做完，而是把接下来要做的内容、实施顺序、交付物和验收方法写清楚，便于审核后执行。

---

# 1. 计划目标

本轮计划聚焦在两件事：

1. **让现有试卷/讲义类 DOCX 的输出更稳定、更像文档结构**
2. **为后续表格能力和中间结构层打基础**

不追求一步到位支持所有 Word 特性。
当前优先考虑真实样本里已经反复出现的能力缺口。

---

# 2. 本轮建议范围

## 2.1 建议纳入本轮的内容

### A. 编号与列表结构恢复
目标：
- 解析 `numbering.xml`
- 恢复有序/无序列表
- 支持多级编号基础结构
- 对高置信纯文本编号做保守恢复（如 `1、` / `（1）` / `1.`）
- 让试卷中的题号、章节条目、分项更接近原结构

### B. 试卷题型结构恢复
目标：
- 识别类似：
  - 计算题（每题…）
  - 填空题（每题…）
  - 解答题（每题…）
- 输出成更明确的结构块，而不是普通段落

### C. 诊断能力继续增强
目标：
- 让坏 OLE / 回退 / 跳过在 report 中更清晰
- 尽量减少底层噪音日志对正常输出的干扰

### D. 为表格支持做结构预埋
目标：
- 先不做完整表格渲染
- 但先抽出统一 `Block` / 中间结构方向
- 为后续 Table AST 留出接口和组织方式

---

## 2.2 不建议纳入本轮的内容

这些建议放到下一阶段：

- 完整表格渲染
- rowspan / colspan 全功能支持
- 图题 / 表题绑定
- 脚注 / 尾注
- 超链接 / 交叉引用
- 完整模板系统
- 多输出格式（HTML / Markdown / JSON）

原因：
- 本轮如果同时做这些，会把范围拉得过大
- 当前最直接影响真实样本质量的，是编号/列表，而不是复杂表格

---

# 3. 分阶段计划

---

## Phase 1：编号 / 列表能力

### 要做什么
1. 解析 `word/numbering.xml`
2. 读取：
   - `abstractNum`
   - `num`
   - `lvl`
   - `numFmt`
   - `lvlText`
3. 在正文段落中识别带 `w:numPr` 的段落
4. 建立基础列表中间结构：
   - list type
   - level
   - items
5. LaTeX 先支持：
   - `enumerate`
   - `itemize`

### 预期结果
- 原来零散的编号段落，能够输出成连续列表结构
- 简单试卷题号结构更自然
- 题号中间夹公式/图片/说明的场景，不会轻易断裂

### 风险点
- Word 编号系统本身比较绕
- 一些试卷文档可能把“题号”混在普通段落和编号样式之间
- 纯文本编号恢复存在误判风险，尤其是答案/详解/证明区

### 验证方式
#### 单测
- 新增 `numbering.xml` 解析测试
- 覆盖：
  - 单级有序列表
  - 单级无序列表
  - 多级编号

#### 样本验证
用已有真实样本：
- `0407定时训练B...docx`
- `0403周六定时训练A...docx`

重点观察：
- “1. 2. 3.” / `1、` / `（1）` 是否能恢复成结构化列表
- 层级是否正确
- 公式顺序是否没有被破坏
- 答案/详解区是否没有被误加成题号列表

---

## Phase 2：题型结构恢复

### 要做什么
1. 增加一层试卷语义识别
2. 识别典型题型标题：
   - `计算题（...）`
   - `填空题（...）`
   - `解答题（...）`
3. 将其输出为更明确的结构：
   - 可先作为 `\section{}` / `\subsection{}`
   - 或作为自定义 block，后续 renderer 再决定

### 预期结果
- 试卷阅读体验更好
- 文档结构比“标题 + 普通段落”更清楚
- `综合题（第一题...）` 这类标题不再掉进普通列表/正文流

### 风险点
- 不同试卷模板标题写法不统一
- 需要避免误判普通正文

### 验证方式
#### 单测
- 文本模式匹配测试
- 避免误把普通句子识别为题型标题
- 覆盖 `综合题（...）` 等非“每题”型标题

#### 样本验证
观察：
- 是否正确识别试卷的大题分组
- 是否不会误伤正常段落

---

## Phase 3：诊断与报告增强

### 要做什么
1. 细化 OLE 错误原因分类，例如：
   - invalid-ole
   - mtef-open-panic
   - empty-output
   - bad-latex
   - fallback-image
2. report 中补充更清晰字段
3. 减少底层解析库噪音日志外溢
   - 包括 CLI 层对 `eqn` 子链路与 stdlib logger 噪音的统一静默处理
4. 明确区分：
   - converted
   - skipped
   - fallback-image
   - convert-error

### 预期结果
- 用户能直接从 report 看出哪些地方有问题
- 调试坏文件更方便

### 风险点
- 底层第三方库日志不一定容易完全消掉
- 需要避免为了“日志干净”而隐藏关键错误

### 验证方式
#### 单测
- 错误分类函数测试
- report 输出结构测试

#### 样本验证
- 坏 OLE 样本中 warning 数量和分类是否合理
- report 是否能解释为什么 fallback

---

## Phase 4：中间结构预埋（轻量）

### 要做什么
不在本轮彻底重构，但先做最小预埋：

1. 引入统一 `Block` 概念
2. 在设计上把这些类型独立出来：
   - Paragraph
   - Heading
   - Math
   - Image
   - List
3. 新增内部注释和代码组织，避免后续继续把所有逻辑堆在 `docx.go`

### 预期结果
- 为后续 Table AST 做准备
- 降低以后大改时的迁移成本

### 风险点
- 如果做得太重，会把本轮范围拉大
- 如果做得太轻，又起不到基础作用

### 验证方式
- 以代码结构可维护性为主，不追求用户侧立刻可见效果
- 通过评审确认接口和数据组织是否合理

---

# 4. 暂缓到下一轮的计划

## 表格支持（下一轮主项）
本轮不建议完整实现表格，但下一轮建议把它作为重点：

### 目标
- 解析 `w:tbl`
- 建立 Table AST
- 支持基础列宽
- 支持 `colspan`
- 再逐步支持 `rowspan`

### 为什么放下一轮
- 表格的复杂度明显高于编号/列表
- 直接做很容易把范围失控
- 当前真实样本里，编号/列表对最终效果提升更直接

# 6. 怎么检验

这个项目不能只看“程序有没有退出”，必须分层验收。

---

## 6.1 基础验收：不崩
### 验收标准
- 正常样本不崩
- 坏样本不崩
- 对于坏 OLE，允许 fallback，但不能 panic

### 验证命令
```bash
go test ./...
go run . --wordDocx sample.docx --report
```

---

## 6.2 结构验收：列表/题型恢复
### 验收标准
- 带编号段落能形成结构化输出
- 试卷大题分组比当前版本更清楚
- 标题、题号、正文、公式顺序不乱

### 验证方法
对比 `.tex` 输出：
- 是否出现 `enumerate` / `itemize`
- 是否出现明确的大题结构
- 是否仍保留原始题目顺序

---

## 6.3 诊断验收：report 可解释
### 验收标准
- report 里能看出：
  - 成功公式数量
  - fallback 数量
  - warning 数量
  - 失败原因类型
- 不再只有模糊报错

### 验证方法
检查 `.report.json`：
- summary 是否完整
- equations 条目是否合理
- warnings 是否能对上真实异常点

---

## 6.4 回归验收：真实样本稳定
### 样本集合
至少包括：
- `0407定时训练B...docx`
- `0403周六定时训练A...docx`

### 验收标准
- 0407：正常输出，结构优于当前版本
- 0403：不崩，且 fallback 行为稳定

---

# 7. 本轮完成标志

当以下条件同时满足时，认为本轮可以结束：

1. `go test ./...` 通过
2. 真实样本转换不崩
3. 编号/列表恢复已经落地
4. 试卷题型结构较当前版本明显改善
5. report 比当前更可解释
6. 文档更新完成

---

# 8. 我建议的执行顺序

建议严格按这个顺序来做：

1. **先做 numbering 解析**
2. **再做列表结构恢复**
3. **再做题型结构识别**
4. **再增强 report**
5. **最后做轻量 Block 预埋**

不要反过来。原因是：
- numbering / list 是当前收益最高的部分
- 先把用户最能感知到的东西做好
- 再去整理内部结构

---

# 9. 审核点

请重点审核这些问题：

1. 本轮范围是否合适
2. 是否现在就做表格，还是下一轮做
3. 题型结构识别应该直接输出 section，还是先做成内部 block
4. report 需要多详细
5. 是否接受先做“结构正确优先，样式保真其次”的策略

---

# 10. 建议结论

我的建议是：

## 本轮就做
- 编号 / 列表
- 题型结构
- report 增强
- 轻量结构预埋

## 下一轮再做
- 表格 AST
- 表格渲染
- 更复杂的文档语义支持

这是当前性价比最高的路线。
