# 1. 项目定位

`docxtolatex` 是一个 **DOCX → LaTeX 转换工具**，当前重点场景是：

1. 解析 Word 文档中的 **MathType OLE 公式**
2. 解析 Word 原生 **OMML 公式**
3. 按文档顺序输出正文、公式、图片到 `.tex`
4. 在公式损坏时尽量做 **图片回退**，而不是直接失败

当前它更接近：

> 一个面向“包含大量公式的试卷/讲义/题库文档”的 LaTeX 提取器

而不是一个完整意义上的“高保真 Word 文档排版还原器”。

---

# 2. 当前设计目标

当前阶段最重要的三个目标是：

## 2.1 不崩
- 坏 OLE 不应该让程序 panic
- 部分损坏文档也应该尽量产出结果
- 能转的公式继续转，坏的公式尽量回退图片

## 2.2 可诊断
- 需要知道哪些公式成功了
- 哪些公式失败了
- 为什么失败
- 回退成了什么

## 2.3 可扩展
- 先有基础文档结构恢复能力（标题、引用、图片输出策略）
- 后续再扩展编号、列表、表格、图题、脚注等结构

# 4. 运行入口与核心流程

## 4.1 CLI 入口
入口在 `main.go`。

当前支持：
- `--filepath, -f`：转换单个 OLE 对象文件
- `--wordDocx, -w`：转换整份 `.docx`
- `--output, -o`：指定输出目录
- `--config, -c`：指定 JSON 配置文件
- `--report, -r`：输出 JSON 转换报告

## 4.2 主流程
`docx.Converter.Convert()` 是当前核心入口。

主流程大致是：

1. 打开 `.docx`（zip）
2. 读取 `word/document.xml`
3. 读取 `word/_rels/document.xml.rels`
4. 读取 `word/styles.xml`（用于 style id → style name 映射）
5. 线性扫描 XML token
6. 识别：
   - 段落
   - 文本
   - OLEObject
   - OMML
   - 图片
7. 组装 `.tex`
8. 可选输出 `.report.json`

目前主流程仍以 **“解析后再统一渲染段落块”** 为主，已经不再是完全线性直写字符串：
- 段落会先收集成轻量 `paragraphBlock`
- 再统一做列表提升、题型结构判断和 LaTeX 渲染

但它还没有真正独立成完整 AST/IR，当前只能算 **轻量中间结构预埋**。

# 5. 当前已经具备的能力

## 5.1 OLE 公式转换
- 支持从 `word/embeddings/oleObject*.bin` 解析 MathType OLE
- 支持按正文顺序输出
- 支持缓存同一 target 的转换结果

## 5.2 OMML 公式转换
- 支持 Word 原生 OMML 节点转换
- 支持基础分式、上下标、根式、定界符、矩阵、对齐等结构

## 5.3 基础样式与结构恢复
当前已支持：
- `Title`
- `Subtitle`
- `Heading 1`
- `Heading 2`
- `Heading 3`
- `Quote`
- `Intense Quote`

并且已经支持：
- `styles.xml` 中的 `styleId` 映射到样式名
- 例如文档里 `w:pStyle w:val="2"`，可以恢复成 `heading 1`
- `numbering.xml` 解析与 Word 原生列表恢复
- 一部分高置信纯文本编号恢复（如 `1、` / `（1）` / `1.`）
- 试卷题型标题恢复（如 `计算题（...）` / `综合题（...）`）

## 5.4 配置能力
通过 `config.example.json` / `--config` 已支持：
- 文档类
- LaTeX 包列表
- 样式映射模板
- 图片输出模式
- 报告开关和路径

## 5.5 图片处理
- 提取图片到 `img/`
- 图片文件名带哈希，避免同名覆盖
- 支持图片占位输出模式配置

## 5.6 报告能力
当前可输出 `.report.json`，包含：
- 段落数
- 公式总数
- OLE 成功数
- OMML 成功数
- 图片回退数
- 提取图片数
- 列表项计数
- 试卷题型标题计数
- warnings
- 每个公式的索引、来源、状态、原因

## 5.7 坏 OLE 容错
这是目前非常重要的一项改动：

过去：
- 坏 OLE 可能触发底层解析 panic
- 整份转换直接中断

现在：
- `eqn/convert.go` 对 OLE 转换做了 `recover()` 保护
- 遇到坏 OLE 时不会直接崩
- `docx/docx.go` 中会尽量回退为预览图
- 并在 report/warnings 中记录失败原因

# 6. 当前已知现实样本结论

项目已经拿真实样本做过验证，而不是只看 README。

## 6.1 样本：0407 定时训练 B
特点：
- 大量 OLE
- 结构相对规整
- 适合验证标题样式恢复

结果：
- 标题识别正常
- OLE 主链路可用
- 说明当前“正常文件路径”基本通顺

## 6.2 样本：0403 定时训练 A
特点：
- 大量 OLE
- 含有一批坏 OLE payload

结果：
- 两个用户上传的 0403 文件实为同一文件副本（SHA256 相同）
- 原始版本会因坏 OLE panic
- 修复后可完成转换
- 大致表现为：
  - 一部分公式正常转换
  - 一部分公式回退图片
  - 整份文档仍能输出 `.tex`

结论：
- “坏 OLE 容错”是必须长期保留的能力
- 当前工具对试卷类文档已经具备一定实用性
- 编号/题型结构恢复已经明显改善
- 但纯文本题号仍然属于启发式恢复，表格和更复杂结构仍然不够

# 7. 当前架构问题

## 7.1 最大问题：还没有真正的中间结构层
当前虽然已经引入轻量 `paragraphBlock` 和统一渲染阶段，但核心解析仍主要集中在 `docx.go` 中。

这会导致：
- 结构恢复逻辑仍然容易继续堆进 `docx.go`
- 列表、题号、表格等能力还没有真正沉到稳定 IR 层
- 列表、表格、图题、脚注等功能难以优雅扩展
- 渲染和解析耦合严重

## 7.2 编号/列表能力不足
虽然文档里有 `numbering.xml`，但当前还没有系统解析：
- 有序列表
- 无序列表
- 多级编号
- 试卷题号结构

因此像：
- 计算题
- 填空题
- 解答题
- 题目编号

很多还只是“普通段落”，而不是结构化输出。

## 7.3 表格能力尚未建立
目前尚未系统支持 `w:tbl`。
后续如果做表格，必须建立独立的表格中间结构，不建议直接从 XML 现场拼 LaTeX。

## 7.4 诊断虽然有了，但还不够精细
目前已有 report 和 warnings，但后续仍建议加强：
- 更细的错误分类
- 更干净的底层日志处理
- 更明确的 fallback 统计

# 8. 开发原则

后续继续改这个仓库时，建议遵守这些原则：

## 8.1 不要为了“高保真”牺牲稳定性
优先级应该是：
1. 不崩
2. 有结果
3. 结构尽量保真
4. 样式再精修

## 8.2 失败必须可见
如果某个内容没法正确转换：
- 不要静默吞掉
- 要么回退
- 要么记录 warning/report

## 8.3 解析与渲染要逐步分离
目标方向是：
- parse
- normalize
- render

即使暂时还没完全拆开，也应避免继续加重耦合。

## 8.4 真实样本优先于想象中的边界条件
这个项目本质上是工程工具。
所以优先级应依据：
- 用户真实上传文档
- 常见试卷/讲义结构
- 高概率坏文件模式

而不是先为了理论上所有 DOCX 特性做大而全支持。

---

# 9. 后续推荐演进路线

## Phase 1：稳定性与诊断增强
- 收敛底层 OLE 噪音输出
- 细化 OLE 失败类型
- 加更多坏文件样本回归测试
- 明确区分：成功、回退、跳过、失败

## Phase 2：编号与列表结构恢复
- 解析 `numbering.xml`
- 恢复有序/无序列表
- 恢复试卷题号结构
- 让“计算题/填空题/解答题”更结构化

## Phase 3：引入中间结构层
- 定义 `Block` 抽象
- 把 Paragraph / Heading / Image / Math / Table 纳入统一 IR
- 将 parse/normalize/render 分离

## Phase 4：表格能力
- 解析 `w:tbl`
- 建立 Table AST
- 支持 colspan/rowspan
- 再做 LaTeX renderer

## Phase 5：高级文档结构
- 图题 / 表题
- 脚注 / 尾注
- 超链接 / 书签 / 交叉引用
- 模板化输出（article/report/exam 等）

# 11. 当前最值得优先做的事

如果只选一件事继续往下做，建议优先做：

## **编号 / 列表结构恢复**
原因：
- 真实试卷文档里出现频率高
- 对最终可读性影响非常大
- 工程复杂度低于表格
- 能直接让现有样本输出质量提升一个台阶

---

# 12. 不建议现在就做的事

短期内不建议优先投入：
- 复杂嵌套表格的高保真支持
- 完整 Word 样式体系全覆盖
- 精细排版级别的 LaTeX 微调
- 一开始就支持所有文档特性

这些事情会显著抬高复杂度，但对当前主要样本收益不一定最高。

---

# 13. 当前结论

`docxtolatex` 已经从一个“只会简单抽取公式”的工具，走到了一个更像样的工程化转换器雏形：

- 有配置
- 有报告
- 有基础结构恢复
- 有坏文件容错
- 有真实样本验证

但它还没到“成熟的文档转换器”阶段。

下一步正确方向不是继续堆 if/else，而是：

> 在保持现有可用性的基础上，逐步引入结构层，优先补齐编号/列表，再进入表格。
